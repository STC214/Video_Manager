# 视频归档工具测试用例

## 目标

本文档记录当前自动化测试与人工验收必须覆盖的核心场景，用于验证扫描、分组、dry-run、移动、重名处理、清理和恢复能力。

## 基础扫描

### 资源管理器式目录选择

- 点击源目录或目标目录的 `浏览...` 后，应打开 Windows `IFileOpenDialog` 文件夹选择器。
- 对话框应包含地址栏、侧边栏、搜索框和资源管理器文件夹视图。
- 当前输入路径有效时，应尝试从该路径打开。
- 取消对话框后主窗口应继续响应，且不修改输入框内容。

### 0 个视频

- 源目录存在，但没有可识别视频文件。
- 扫描结果应显示视频总数为 0。
- 不应生成移动计划。
- `开始移动` 应保持不可用。

### 1 个视频

- 应生成 1 个 `Episode_001`。
- 两层模式应生成 `Season_001/Episode_001`。
- 三层模式应生成 `Arc_01/Season_001/Episode_001`。

### 扩展名大小写

- 输入包含 `.mp4`、`.MP4`、`.MkV`。
- 应全部按视频文件识别。

### 非视频文件

- 输入包含 `.txt`、`.jpg`、`.nfo`。
- 默认不移动非视频文件。
- 扫描统计应显示非视频文件数量。

## 分组边界

### 刚好一个叶目录

- 两层模式输入 25 个视频。
- 应只生成 `Episode_001`，其中包含 25 个视频。

### 超出一个叶目录

- 两层模式输入 26 个视频。
- 应生成 `Episode_001` 和 `Episode_002`。
- `Episode_001` 包含 25 个视频，`Episode_002` 包含 1 个视频。

### 大规模三层示例

- 三层模式输入 7429 个视频。
- 参数为每个 `Episode` 最多 30 个视频，每个 `Season` 包含 5 个 `Episode`，每个 `Arc` 包含 10 个 `Season`。
- 应生成 248 个 `Episode`、50 个 `Season`、5 个 `Arc`。
- 最后一个 `Episode_248` 应包含 19 个视频。

## 归档容量计算器

### 三层容量计算

- 目标文件总数为 7429。
- 结果目录层数为 3。
- 第 1 层目录名为 `Arc`，文件夹数为 5。
- 第 2 层目录名为 `Season`，文件夹数为 10。
- 第 3 层目录名为 `Episode`，文件夹数为 5。
- 叶目录文件数为 30。
- 应显示需要叶目录数量为 248。
- 应显示最后叶目录为 `Arc_05/Season_050/Episode_248`。
- 应显示最后叶目录文件数为 19。
- 应显示每层实际目录数量为 5、50、248。
- 应显示当前配置最大容量为 7500。

### 动态层级输入框

- 用户把结果目录层数从 2 改为 3。
- UI 应新增第 3 层目录名输入框和对应容量输入框。
- 原有第 1 层和第 2 层输入值应保留。

### 命名预案

- 用户选择 `Volume / Part / Batch` 预案。
- 三层目录名应自动填入 `Volume`、`Part`、`Batch`。
- 用户手动修改某一层目录名后，计算结果应使用手动名称。

## 排序稳定性

### 修改时间排序

- 输入来自多个子目录。
- 应按最后修改时间从早到晚生成移动计划。
- 越早的文件应进入编号越小的叶目录。
- 修改时间相同时，应按相对路径升序，再按文件名升序。
- 同一输入重复生成 dry-run，目标路径应完全一致。

### 混乱目录名

- 输入目录包含无意义、重复或混乱命名的多层子目录。
- 程序不应根据目录名推断分类。
- 程序应递归扫描视频文件，并只在修改时间相同时把目录名用于稳定排序和来源追踪。

### 已归档目录排除

- 目标目录位于源目录内部。
- 扫描时应排除目标根目录本身。
- 重复运行不应把已归档视频再次纳入新计划。

## 目标目录和重名

### 目标目录不存在

- 移动前应自动创建目标目录。
- 创建失败时应记录错误并跳过对应文件。

### 目标目录已存在

- 应允许继续生成 dry-run。
- dry-run 应显示目标冲突数量。
- 不应覆盖已有文件。

### 同名文件冲突

- 两个不同源目录中存在同名 `a.mp4`，并被分配到同一个叶目录。
- 第一个文件保留 `a.mp4`。
- 第二个文件应命名为 `a_dup001.mp4`。

### 多重同名冲突

- 目标目录中已有 `a.mp4`、`a_dup001.mp4`。
- 新文件应命名为 `a_dup002.mp4`。

## Dry-run 和日志

### Dry-run 不移动文件

- 生成 dry-run 后，源文件仍应留在原位置。
- dry-run 阶段只计算计划，不创建目标目录。
- 目标目录只在正式移动时创建。
- dry-run TSV 写入程序目录 `data/dry-runs/`。

### Dry-run 摘要

- dry-run 应显示视频总数、总移动数量、目标目录数量。
- dry-run 应显示预计 `Arc`、`Season`、`Episode` 数量。
- dry-run 应显示最后一个叶目录预计文件数。
- dry-run 应显示重名冲突、不可访问文件和计划清理空目录数量。

### 结构预览

- dry-run 应能展示目标目录树摘要。
- 每个 `Episode` 节点应显示预计文件数量。
- 大量节点时 UI 可以只展开前几组，但完整数据应可导出或滚动查看。

### Dry-run TSV

- 导出的 TSV 至少包含：

```text
status	source	target	size	conflict	error
```

- 路径中包含空格和中文时，导出内容应保持可读。

### 运行日志

- 正式移动时应在运行清单中记录每个文件的结果。
- UI/应用日志显示时间和节流后的任务进度；运行清单保存完整源路径、目标路径、结果和错误信息。

## 移动安全

### 同卷移动

- 同一卷内应优先使用 rename / move。
- 成功后源路径不存在，目标路径存在。

### 跨卷移动

- 跨卷时应 copy + verify + delete。
- 首版校验标准为文件大小一致。
- 校验失败时不删除源文件。

### 移动失败

- 某个文件被占用或权限不足。
- 应记录失败并继续处理其他文件。
- 最终状态应为 `CompletedWithErrors`。

### SMB 映射盘

- 源目录或目标目录位于软路由 / NAS 通过 SMB 映射的盘符，例如 `Z:\Videos`。
- 扫描、dry-run 和正式移动都应正常工作。
- 如果直接 move 失败，应退化为 copy + verify + delete。

### SMB UNC 路径

- 源目录或目标目录使用 UNC 路径，例如 `\\router\share\Videos`。
- 路径可访问时应正常扫描和移动。
- 路径不可访问或登录状态失效时，应记录错误并提示用户。

### 网络中断

- dry-run 成功后，正式移动时 SMB 路径中断。
- 程序应重新检查源文件存在性。
- 失败项应记录错误，不应中断整个任务。

### 用户取消

- 移动过程中取消任务。
- 已完成的文件保持已移动状态。
- 未处理文件保持在源目录。
- 日志应记录取消点。

### Dry-run 后源文件变化

- 扫描或 Dry-run 后，源文件大小或最后修改时间发生变化。
- 正式移动时应拒绝按旧计划移动该文件，并保留源文件。
- 用户应重新扫描并生成新的 Dry-run，以保证时间排序仍然正确。

## 清理空目录

### 删除空旧目录

- 移动完成后，源目录中的旧空子目录应被删除。

### 保留非空目录

- 源目录中仍有非视频文件的目录不应删除。

### 保留隐藏文件目录

- 目录中有隐藏文件时不应删除。

### 不删除新目标结构

- 目标目录位于源目录内时，清理过程不应删除新建的目标结构。

## 恢复和重跑

### 失败后重跑

- 第一次运行部分文件失败。
- 第二次运行应能重新扫描剩余文件。
- 已成功移动的文件不应被重复移动。

### 撤销清单

- 每次正式移动应生成运行清单。
- 清单应记录原路径、目标路径、状态和文件大小。
- 当前程序可基于最近一次清单执行反向移动。
- 撤销前应核对归档文件大小；文件已被替换或修改时不得移动回源路径。
- 完整撤销可重复执行，已恢复且大小一致的文件按完成处理。
